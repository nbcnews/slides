Why SPA/JS?

Enhanced UX and Dev/Designer Workflow

  • Lines blurring between native, traditional web applications.
  • Flexibility in design (transitions, animations)
  • Continuous, transparent consumption
  • Removes need to download HTML payload between pages
  • Backend flexibility (relies on JSON - server platform agnostic)

Why Ember?

Initial Analysis

  • Opinionated architecture can allow transfer of knowledge between developers in teams, brands
  • Focus on the URL (routes!)
  • Documentation (aids in learning curve)
  • Handlebars
  • Tooling
  • Browser compatibility
  • Community (Seattle, NYC)
  • Designers proficient in SASS/CSS, HTML, jQuery. Unencumbered access to markup essential.
  • URL is of utmost importance.
  • Sharing, navigation (links, back button).
  • Grunt, Node

Challenges

  • Ads (other third party scripts)
  • Custom server rendering for SEO
  • Modifying destroyed views - understanding run loop
  • Memory awareness
  • Another level of complexity to shape API
  • Ads require custom management
  • Third party scripts modifying the URL of iframes
  • Checking willDestroy before manipulating
  • Memory is reset per page transition in traditional apps. Equivalent to restarting computer for C++ apps
